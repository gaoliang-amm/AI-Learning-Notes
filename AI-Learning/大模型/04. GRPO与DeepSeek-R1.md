# GRPO 与 DeepSeek-R1 复刻

## 一、DPO 微调：直接偏好优化

### 1.1 DPO 目标函数

DPO（Direct Preference Optimization）核心洞察：**奖励函数与最优策略之间存在直接映射**，无需单独训练奖励模型。

偏好数据集格式：`{prompt, chosen, rejected}`

$$J_{DPO}(\pi_\theta; \pi_{ref}) = \mathbb{E}_{(y_w, y_l, x) \sim \mathcal{D}} \left[ \log \sigma \left( \beta \left( \log \frac{\pi_\theta(y_w|x)}{\pi_{ref}(y_w|x)} - \log \frac{\pi_\theta(y_l|x)}{\pi_{ref}(y_l|x)} \right) \right) \right]$$

**直觉理解**：DPO 让模型"喜欢的回答概率越来越高，讨厌的回答概率越来越低"。$\beta$ 控制偏离参考模型的程度——越大越保守，越小越激进。**为什么 margin 越大越好？** 类比"好回答的概率应该远高于差回答"——如果一个回答只是稍微好一点点，模型很难区分；但如果好的概率被推到远高于差的，决策边界就清晰了，泛化能力也更强。

隐式奖励形式：
$$r^*(x,y) = \beta \log \frac{\pi_\theta(y|x)}{\pi_{ref}(y|x)} + \text{const}(x)$$

### 1.2 DPO 存在的问题

| 问题 | 表现 | 后果 |
|------|------|------|
| 过拟合 | $\frac{\pi_\theta(y_w)}{\pi_\theta(y_l)}$ 分子→1、分母→0 | 参考模型正则化失效，探索能力丧失 |
| 偏好循环 | 数据集中出现 $A \succ B$ 且 $B \succ A$ | 模型学到矛盾信号，无法收敛 |
| 正负例模糊 | 人类都难以区分的回答差异 | 模型学不到有效信号 |

DPO 不能训练太多轮！Epoch 过多会彻底过拟合。**直觉类比**：就像考试只刷题不理解，遇到新题就不会了——模型死记硬背偏好数据中的"答案模式"，对训练集中没出现过的 prompt 就无法正确判断偏好。

### 1.3 SFT + DPO 实战流程（Qwen3-0.6B 案例）

```
预训练模型 → SFT（1 epoch，只对答案部分算损失）→ DPO（偏好数据微调）
```

**SFT 关键细节**：
- 只对 Assistant 回答部分计算损失，忽略 User 问题部分
- 使用掩码：`shift_logits` 和 `shift_labels` 对齐后，将问题部分的 loss 设为 -100（PyTorch 忽略标记）
- 1 epoch 足够，防止灾难性遗忘

**DPO 训练要点**：
- 同时计算偏好/拒绝两条数据的前向传播
- 参考模型 $\pi_{ref}$ 只做前向传播，不计算梯度
- 损失：`loss = -log(sigmoid(β * (log_prob_chosen - log_prob_rejected)))`
- 使用余弦退火学习率 + 线性预热

## 二、PPO 微调 LLM：复刻 InstructGPT

> PPO 的数学原理（裁剪机制、重要性采样等）详见 [[03. PPO与RLHF对齐]]

### 2.1 InstructGPT 三步骤

1. **SFT**：用人工标注的 prompt→response 对微调，仅对回答部分计算损失
2. **RM（奖励模型）**：同一 prompt 生成多个回复，人工排序后训练分类模型学习优劣评价
3. **PPO**：用奖励模型打分反馈更新策略，同时用 KL 散度约束防止偏离 SFT 模型

### 2.2 KL 惩罚项精解

$$R_t = \begin{cases} -\beta\log\frac{\pi_\theta(y_t|x,y_{<t})}{\pi_{ref}(y_t|x,y_{<t})} & t < T \\ R_{RM} - \beta\log\frac{\pi_\theta(y_T|x,y_{<T})}{\pi_{ref}(y_T|x,y_{<T})} & t = T \end{cases}$$

**为什么 KL 惩罚是逐 token 的？** 因为 LLM 是自回归生成，每一步都有可能"说错话"。如果只有结尾的 RM 打分，中间的乱码 token 不会被惩罚。

**KL 惩罚的两个作用**：
- 防止 Reward Hacking：没有 KL 约束，模型会找到奖励模型的漏洞，生成高分但人类读不通的乱码
- 保持语言流畅性：锚定 SFT 模型，保证输出是"人话"

**直觉类比**：KL 惩罚像是给模型装了一个"道德底线"——模型想尽办法拿高分（最大化奖励），但 KL 散度时刻提醒它"别为了高分说胡话"，偏离参考模型太远就要扣分，从而在"拿高分"和"说人话"之间取得平衡。

### 2.3 稀疏奖励 vs 稠密奖励

| 奖励类型 | 来源 | 发放时机 | 作用 |
|---------|------|---------|------|
| 稀疏奖励 | RM 打分 | 仅 `<eos>` 时 | 引导方向（"人类喜欢什么"） |
| 稠密奖励 | KL 惩罚 | 每一步 token | 约束边界（"不要乱说话"） |

PPO 的 GAE（广义优势估计）将最后一步的稀疏奖励沿时间轴回传，让中间 token 也能获得梯度信号。

### 2.4 PPO 实战关键洞察（电商评论场景）

**任务**：微调 GPT-2，让它写正向电商评论

**ActorCritic 架构合并**：演员和评论家共享 GPT-2 底座权重，只在输出头分叉——Actor 输出 token 概率，Critic 输出状态价值。共享权重减少显存，加速收敛。

**奖励缩放**：RM 输出 $(0,1)$ 分数，需映射到 $(-1,1)$：`score = 2 * rm_score - 1`

**训练循环要点**：
- 每条轨迹（prompt + 补全）通过 PPO 更新 4 次
- 使用 32 条轨迹批量更新策略模型
- 评估方法：对比微调前后模型的 RM 平均分

## 三、GRPO 微调：复刻 DeepSeek-R1

> GRPO 的基础原理和目标函数详见 [[03. PPO与RLHF对齐]]

### 3.1 DAPO：GRPO 的改进

DAPO（字节提出）相比原始 GRPO 的两项核心改进：

| 改进 | 原始 GRPO | DAPO |
|------|----------|------|
| 策略梯度 | token 级加权 | **token 级等权**（每个 token 同等重要） |
| KL 散度 | 显式损失项 $-\beta D_{KL}$ | **完全移除**（省显存，无需 $\pi_{ref}$） |

当每次迭代只更新一次策略时（$\theta = \theta_{old}$），裁剪不生效，目标函数梯度退化为：

$$\nabla_\theta \log\pi_\theta(a_{i,j}[t]) \cdot A_{i,j}[t]$$

即原始策略梯度 × 组相对优势。

**直觉类比**：GRPO 的组相对优势就像全班考试排名——不需要知道每个学生的绝对分数，只要在一组（比如 8 个）回答中算出均值和标准差，就能判断哪个回答"相对更好"。得分高于组平均的就是正优势，低于平均的就是负优势。这种"组内比"的方式天然消除了奖励尺度的不确定性。

### 3.2 CountDown Task：24 点游戏实战

**任务**：给定数字列表和目标值，用 `+ - * /` 生成等于目标值的表达式

**奖励函数设计**（满分 1.1）：

| 奖励 | 条件 | 分值 |
|------|------|------|
| 格式奖励 | 正确使用 `<think>` + `<answer>` 标签 | 0.1 |
| 答案奖励 | 使用了所有数字且计算正确 | 1.0 |

**训练流程**：
```
每 batch 32 个问题 → 每个问题采样 8 条回答 → 并行推理
→ 计算组内归一化优势（均值/标准差）→ 梯度更新
```

**为什么用 GRPO 而不是 PPO？** 这类推理任务需要探索多种解题路径（比如先乘后加，还是先加后乘），GRPO 的组采样天然适合——每个问题同时生成 8 条不同路径的回答，组内直接比较优劣，无需额外训练 Critic 网络来估计价值函数，既省显存又能高效探索解题空间。

## 四、GRPO 应用场景

### 4.1 医疗思维链推理

**数据集混合**：
- PubMedQA（70%）：临床问答 yes/no/maybe
- GSM8K：数学应用题，保持数学推理能力
- Health Benchmarks：50+ 道医学专业选择题

**多奖励函数**：格式奖励（strict/soft/xmlcount）、整数格式奖励、答案正确性奖励

### 4.2 Text-to-SQL

底座模型：Qwen2.5-Coder-7B-Instruct

**三种奖励校验**：
1. 正则表达式校验 SQL 格式
2. 实际执行 SQL 校验正确性（sqlite3）
3. 调用外部 LLM（DeepSeek/GPT）评估思维链质量

## 五、关键要点总结

| 维度 | DPO | PPO | GRPO |
|------|-----|-----|------|
| 是否需要 RM | 否 | 是 | 否 |
| 是否需要 Critic | 否 | 是 | 否 |
| KL 散度 | 隐式（$\pi_{ref}$） | 显式逐 token | 可选（DAPO 移除） |
| 显存需求 | 低 | 高（Actor+Critic+RM） | 中（仅 Actor） |
| 适合场景 | 偏好数据充足 | 通用对齐 | 推理能力提升 |
| 代表工作 | 开源 LLM 指令微调 | InstructGPT/ChatGPT | DeepSeek-R1 |
