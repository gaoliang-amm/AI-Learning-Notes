# RAG 技术全解

> RAG（Retrieval-Augmented Generation）是将外部知识检索与大模型生成能力结合的技术范式，解决 LLM 知识过时、幻觉、无法获取私域数据等问题。

---

## 一、为什么需要 RAG

### LLM 的固有缺陷

| 问题 | 表现 | RAG 如何解决 |
|------|------|-------------|
| **知识截止** | 训练数据有时间边界，无法获取最新信息 | 实时检索外部知识库 |
| **幻觉** | 编造不存在的事实、引用、数据 | 以检索结果为依据生成，可溯源 |
| **私域知识缺失** | 无法获取企业内部文档、个人数据 | 接入私有知识库 |
| **上下文长度限制** | 无法将所有知识塞入 Prompt | 按需检索最相关的片段 |
| **不可解释** | 无法说明答案来源 | 提供引用来源，支持追溯 |

### RAG vs 微调 vs 长上下文

| 方案 | 适用场景 | 优势 | 劣势 |
|------|---------|------|------|
| **RAG** | 知识频繁更新、需要溯源 | 实时性强、成本低、可解释 | 检索质量影响生成质量 |
| **微调** | 特定风格/格式/领域任务 | 深度适配、推理效率高 | 成本高、知识更新需重新训练 |
| **长上下文** | 文档量小（<100K token） | 简单直接 | 成本高、存在 Lost-in-the-Middle |

> **实践建议**: 三者并非互斥，生产中常组合使用（如 RAG + 微调）。

## 二、RAG 核心架构

### 基础 RAG 流程

```
用户提问
   ↓
┌─────────────┐
│ 1. Query 处理  │  ← 查询改写 / 扩展 / 分解
└─────┬───────┘
      ↓
┌─────────────┐
│ 2. 检索 Retrieval │  ← 向量检索 / 关键词检索 / 混合检索
└─────┬───────┘
      ↓
┌─────────────┐
│ 3. 重排序 Rerank │  ← Cross-Encoder 精排
└─────┬───────┘
      ↓
┌─────────────┐
│ 4. 生成 Generation │  ← LLM 基于检索结果生成答案
└─────┬───────┘
      ↓
   最终回答（附引用来源）
```

### 离线索引阶段（Indexing）

```
原始文档
   ↓
┌─────────────┐
│ 文档解析 Parsing │  ← PDF/Word/HTML/Markdown → 纯文本
└─────┬───────┘
      ↓
┌─────────────┐
│ 文本分块 Chunking │  ← 按语义/固定长度/递归分割
└─────┬───────┘
      ↓
┌─────────────┐
│ 向量化 Embedding │  ← 文本 → 高维向量
└─────┬───────┘
      ↓
┌─────────────┐
│ 存入向量数据库    │  ← FAISS / Milvus / Pinecone / ChromaDB
└─────────────┘
```

## 三、文档解析（Document Parsing）

### 常见文档格式与工具

| 格式 | 推荐工具 | 说明 |
|------|---------|------|
| **PDF** | PyMuPDF、pdfplumber、Unstructured | 复杂排版需 OCR 辅助 |
| **Word** | python-docx、Unstructured | 结构化程度高 |
| **HTML** | BeautifulSoup、Trafilatura | 需去除导航、广告等噪声 |
| **Markdown** | 直接解析 | 结构最清晰 |
| **表格** | Camelot、Pandas | 需特殊处理行列关系 |
| **图片/扫描件** | PaddleOCR、Tesseract | OCR 提取文字 |

### 多模态文档处理

对于包含图表、公式的复杂文档：
- **表格** → 转为 Markdown 或 JSON 结构化表示
- **图表** → 使用多模态模型（如 GPT-4V）提取描述
- **公式** → 使用 LaTeX 表示或专用 OCR

## 四、文本分块（Chunking）

分块是 RAG 中**最被低估但影响最大**的环节。

### 分块策略对比

| 策略 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| **固定长度** | 按字符/token 数切分 | 简单可控 | 可能切断语义 | 快速原型 |
| **递归分割** | 按层级分隔符递归切分（`\n\n` → `\n` → ` `）| 保持语义完整性 | 需调参 | **通用首选** |
| **语义分块** | 用 Embedding 相似度判断分割点 | 语义最连贯 | 计算成本高 | 高质量需求 |
| **文档结构** | 按标题/章节/段落分割 | 保留文档结构 | 依赖文档格式规范 | 结构化文档 |
| **滑动窗口** | 固定窗口 + 重叠区域 | 保证上下文连续 | 存储冗余 | 需要上下文连贯 |

### Chunking 最佳实践

```python
# LangChain 递归分割示例
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=512,        # 每块最大 token 数
    chunk_overlap=50,      # 重叠区域，保证上下文连贯
    separators=["\n\n", "\n", "。", "；", " ", ""],  # 中文场景分隔符
    length_function=len,
)
chunks = splitter.split_text(document)
```

### 关键参数建议

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| **chunk_size** | 256-1024 tokens | 过小丢失上下文，过大引入噪声 |
| **chunk_overlap** | 10%-20% of chunk_size | 保证边界信息不丢失 |
| **分隔符** | 按语言/格式调整 | 中文用 `。`、`；`，英文用 `. `、`\n` |

## 五、Embedding 向量化

### 主流 Embedding 模型

| 模型 | 维度 | 特点 | 推荐场景 |
|------|------|------|---------|
| **OpenAI text-embedding-3-large** | 3072 | 商业最强之一 | 英文为主 |
| **OpenAI text-embedding-3-small** | 1536 | 性价比高 | 通用 |
| **BGE-M3** | 1024 | 多语言、开源 | **中文首选** |
| **GTE-large** | 1024 | 阿里开源，中文优秀 | 中文 |
| **E5-large-v2** | 1024 | 微软开源 | 英文 |
| **Jina-embeddings-v2** | 768 | 支持8K长文本 | 长文档 |
| **Cohere embed-v3** | 1024 | 多语言商业模型 | 多语言 |

### Embedding 选择原则

1. **中文场景** → BGE-M3 / GTE 系列
2. **英文场景** → OpenAI / E5 / Cohere
3. **多语言** → BGE-M3 / Cohere embed-v3
4. **长文本** → Jina-embeddings-v2
5. **预算有限** → 开源模型本地部署

### 向量维度与性能权衡

```
维度越高 → 语义表达越精细 → 检索越准 → 存储/计算成本越高
维度越低 → 速度越快 → 可能损失语义信息

推荐: 生产环境 768-1536 维，原型验证 384 维
```

## 六、向量数据库

### 主流向量数据库对比

| 数据库 | 类型 | 特点 | 适用场景 |
|--------|------|------|---------|
| **FAISS** | 库（Meta开源） | 速度极快、内存索引 | 研究/中小规模 |
| **Milvus** | 分布式数据库 | 支持十亿级向量、云原生 | **大规模生产** |
| **ChromaDB** | 嵌入式数据库 | 轻量、易上手 | 原型/小项目 |
| **Pinecone** | 云服务 | 全托管、免运维 | 快速上线 |
| **Weaviate** | 数据库 | 支持混合搜索、GraphQL | 企业级应用 |
| **Qdrant** | 数据库 | Rust 实现、高性能 | 高性能需求 |
| **pgvector** | PostgreSQL 扩展 | 与现有 PG 集成 | 已有 PG 的团队 |

### 向量索引算法

| 算法 | 原理 | 精度 | 速度 | 内存 |
|------|------|------|------|------|
| **Flat (暴力搜索)** | 逐一比较 | 100% | 最慢 | 高 |
| **IVF** | 聚类后搜索最近的簇 | 高 | 快 | 中 |
| **HNSW** | 分层可导航小世界图 | 很高 | 很快 | 高 |
| **PQ (乘积量化)** | 向量压缩 | 中 | 很快 | 低 |
| **IVF-PQ** | IVF + PQ 结合 | 中高 | 很快 | 低 |

> **推荐**: 中小规模用 **HNSW**（精度高速度快），大规模用 **IVF-PQ**（节省内存）。

## 七、检索策略

### 7.1 向量检索（语义检索）

```python
# 核心流程
query_embedding = embedding_model.encode(query)
results = vector_db.search(query_embedding, top_k=10)
```

- **原理**: 计算查询向量与文档向量的余弦相似度 / 内积
- **优点**: 理解语义，"汽车"能匹配到"轿车"
- **缺点**: 对精确关键词匹配较弱

### 7.2 关键词检索（稀疏检索）

- **BM25** 经典算法，基于词频和逆文档频率
- **优点**: 精确匹配关键词、人名、编号等
- **缺点**: 无法理解语义

### 7.3 混合检索（Hybrid Search）

```
向量检索结果 ──┐
               ├──→ 融合 (RRF / 加权) ──→ 最终结果
关键词检索结果 ──┘
```

**倒数排名融合（Reciprocal Rank Fusion, RRF）**:

```
RRF_score(d) = Σ 1 / (k + rank_i(d))

k = 60 (常数)
rank_i(d) = 文档 d 在第 i 个检索器中的排名
```

> **生产推荐**: 混合检索 + RRF 融合，兼顾语义理解和精确匹配。

### 7.4 查询改写（Query Transformation）

| 方法 | 原理 | 适用场景 |
|------|------|---------|
| **HyDE** | 先让 LLM 生成假设性答案，用答案去检索 | 提升语义匹配 |
| **查询扩展** | 用 LLM 生成多个相关查询，合并检索 | 扩大召回 |
| **查询分解** | 将复杂问题拆分为子问题，分别检索 | 多跳推理 |
| **Step-Back** | 先生成更高层次的抽象问题 | 需要背景知识 |

## 八、重排序（Reranking）

检索返回 top-k 后，用更精确的模型重新排序。

### 重排序模型

| 模型 | 类型 | 特点 |
|------|------|------|
| **Cohere Rerank** | 商业 API | 效果好、易用 |
| **BGE-Reranker-v2-M3** | 开源 | 多语言、中文强 |
| **Cross-Encoder** | Sentence-Transformers | 精度高、速度较慢 |
| **ColBERT** | 延迟交互 | 速度与精度平衡 |

### Reranking 流程

```python
# 伪代码
# 1. 初步检索（向量/关键词）返回 top-50
initial_results = retrieve(query, top_k=50)

# 2. Cross-Encoder 精排，返回 top-5
reranked = cross_encoder.rank(query, initial_results, top_k=5)

# 3. 将 top-5 送入 LLM 生成
answer = llm.generate(query, reranked)
```

## 九、高级 RAG 模式

### 9.1 Naive RAG（基础）

```
Query → Retrieve → Generate
```

最简单的流水线，适合快速验证。

### 9.2 Advanced RAG（进阶）

在 Naive RAG 基础上增加：
- 查询改写 / 扩展
- 混合检索
- 重排序
- 自适应检索（判断是否需要检索）

### 9.3 Modular RAG（模块化）

将 RAG 拆分为可插拔模块：
- **Indexing Pipeline**: 解析 → 分块 → 向量化 → 存储
- **Retrieval Pipeline**: 查询处理 → 检索 → 过滤 → 重排
- **Generation Pipeline**: Prompt 构建 → LLM 生成 → 后处理

### 9.4 Self-RAG（自适应检索）

```
用户提问
   ↓
LLM 判断: 是否需要检索？
   ├─ 不需要 → 直接生成
   └─ 需要 → 检索
         ↓
      LLM 判断: 检索结果是否相关？
         ├─ 不相关 → 重新检索 / 改写查询
         └─ 相关 → 生成答案
               ↓
            LLM 判断: 答案是否有依据？
               ├─ 无依据 → 标注不确定
               └─ 有依据 → 返回答案 + 引用
```

### 9.5 Graph RAG（图增强检索）

- 将文档构建为**知识图谱**（Entity → Relation → Entity）
- 检索时同时查询向量库和知识图谱
- 适合需要**多跳推理**和**关系理解**的场景

### 9.6 Agentic RAG（Agent 驱动检索）

- 用 Agent 动态决定检索策略
- 可调用多种检索工具（向量库、搜索引擎、SQL、API）
- 根据中间结果决定是否继续检索
- 详见 [[Agent与工具调用]]

## 十、RAG 评估

### 评估维度

| 维度 | 指标 | 说明 |
|------|------|------|
| **检索质量** | Recall@K | 前 K 个结果中包含正确答案的比例 |
| **检索质量** | Precision@K | 前 K 个结果中正确结果的比例 |
| **检索质量** | MRR (Mean Reciprocal Rank) | 第一个正确结果的排名倒数 |
| **检索质量** | nDCG | 考虑排序位置的评分 |
| **生成质量** | Faithfulness | 答案是否忠于检索到的上下文 |
| **生成质量** | Answer Relevancy | 答案是否与问题相关 |
| **生成质量** | Context Relevancy | 检索到的上下文是否与问题相关 |
| **端到端** | Correctness | 最终答案是否正确 |

### 评估框架

| 框架 | 特点 |
|------|------|
| **RAGAS** | 专为 RAG 设计，自动化评估 Faithfulness/Relevancy |
| **LangSmith** | LangChain 配套，支持 Trace 和评估 |
| **DeepEval** | 开源，支持多种指标 |
| **TruLens** | 可视化评估，支持反馈函数 |

### RAGAS 核心指标计算

```
Faithfulness = (答案中可被上下文支持的声明数) / (答案中总声明数)

Answer Relevancy = mean(cos_sim(原始问题, 生成的问题))
  → 用 LLM 从答案反推问题，比较与原问题的相似度

Context Precision = Σ (相关文档在 top-k 中的排名位置) / 相关文档总数
```

## 十一、生产实践要点

### 常见踩坑与优化

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 检索到但答案不对 | Chunk 太大，噪声多 | 减小 chunk_size，加 Rerank |
| 检索不到相关内容 | Embedding 模型能力不足 | 换更强的 Embedding，混合检索 |
| 答案编造 | LLM 未严格基于上下文 | Prompt 中强调"仅基于以下内容回答" |
| 多文档冲突 | 知识库中有矛盾信息 | 加时间戳过滤，优先最新文档 |
| 延迟过高 | 检索链路长 | 缓存热门查询，异步检索 |
| 长文档丢失信息 | Chunk 切断关键信息 | 增大 overlap，用语义分块 |

### Prompt 工程建议

```
你是一个专业的问答助手。请严格根据以下参考内容回答问题。

规则：
1. 仅使用参考内容中的信息回答，不要编造
2. 如果参考内容中没有相关信息，请明确说"根据已有资料无法回答"
3. 在答案末尾标注信息来源（文档名 + 页码/段落）
4. 如果多个来源有冲突，请指出并说明

参考内容：
{context}

问题：{query}
```

### 架构选型建议

```
小规模 (<1万文档):
  → ChromaDB + 固定长度分块 + 单一向量检索

中规模 (1万-100万文档):
  → Milvus/Qdrant + 递归分块 + 混合检索 + Rerank

大规模 (>100万文档):
  → Milvus 分布式 + 语义分块 + 混合检索 + Rerank + 缓存

知识密集型 (需要推理):
  → Graph RAG / Agentic RAG
```

## 十三、RAG 发展趋势

1. **Agentic RAG** — Agent 动态编排检索策略，取代固定流水线
2. **Multimodal RAG** — 支持图片、表格、音视频的检索与生成
3. **Graph RAG** — 知识图谱增强，支持复杂推理
4. **Cache-Augmented Generation** — 将常用知识缓存，减少检索延迟
5. **Long-context 替代** — 随着上下文窗口扩大，简单场景可能直接塞入全文
